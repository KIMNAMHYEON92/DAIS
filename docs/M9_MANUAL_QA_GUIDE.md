# DAIS Milestone 9 사용자 교차검증 가이드

이 문서는 개발자가 아닌 테스터도 게임 실행부터 TC-1~TC-5 판정, 결과 저장까지 같은 순서로 재현할 수 있도록 작성한 실행 절차서입니다.

## 1. 검증 범위와 판정 원칙

Milestone 9 화면은 기존 코어 모듈을 실제로 연결한 **결정론적 QA 하네스**입니다. 다음 항목은 브라우저에서 직접 조작하고 눈으로 확인할 수 있습니다.

- NPC 말풍선의 실제 DOM 드래그와 오프셋 판정
- 반박문 키워드 검증과 과열 게이지 `-30%`
- 80%/90%/100% 과열 단계의 시각 효과
- PII 비식별화 후 IndexedDB 적재
- 오프라인 큐 적재와 온라인 복구 시 큐 비우기
- 각 TC의 테스터 판정, 메모, JSON/Markdown 결과 내보내기

다음 항목은 현재 저장소에 실제 모델 파일과 Supabase 엔드포인트가 없으므로 QA 하네스만으로 합격 판정하면 안 됩니다.

- 약 1.5GB WebLLM 모델의 실제 다운로드와 VRAM 마운트
- 실제 콜드/캐시 부팅, 첫 토큰, tokens/sec
- 실제 GPU에서 30턴 진행 후 VRAM OOM 여부
- 실제 Supabase 테이블 수신 여부

이 항목들은 연결된 배포 빌드에서 확인할 때까지 `BLOCKED`로 기록합니다. 하네스의 `PASSED`는 해당 로컬 코어 동작이 통과했다는 뜻이지, 외부 시스템까지 통과했다는 뜻이 아닙니다.

## 2. 테스트 전 준비

### 2.1 필요한 환경

- Windows 10/11
- Node.js 20 이상
- 최신 Chrome 또는 Edge
- 실제 WebGPU KPI 검증 시 RTX 3060급 GPU 또는 Apple M1급 장비
- 명령을 입력할 Windows PowerShell

개인정보 테스트에는 반드시 문서의 가짜 번호와 가짜 이메일만 사용합니다. 실제 주민등록번호, 전화번호, 이메일을 입력하지 마십시오.

### 2.2 처음 한 번 설치

1. Windows 시작 메뉴에서 `PowerShell`을 엽니다.
2. 아래 명령을 한 줄씩 입력합니다.

```powershell
cd C:\Users\user\Documents\DAIS
npm.cmd install
```

3. 설치가 끝나고 오류 없이 PowerShell 입력 프롬프트가 다시 나타나는지 확인합니다.

`node_modules` 폴더가 이미 있고 의존성이 설치된 상태라면 `npm.cmd install`은 생략할 수 있습니다.

### 2.3 게임(QA 검증실) 실행

1. PowerShell에서 다음 명령을 실행합니다.

```powershell
cd C:\Users\user\Documents\DAIS
npm.cmd run dev -- --host 127.0.0.1
```

2. PowerShell에 표시되는 `Local` 주소를 확인합니다. 일반적으로 `http://127.0.0.1:5173/`입니다.
3. Chrome 또는 Edge 주소창에 해당 주소를 입력합니다.
4. 화면 상단에 `Milestone 9 // QA 검증실`이 보이는지 확인합니다.
5. 테스트가 끝날 때까지 PowerShell 창을 닫지 않습니다.

실행을 종료하려면 PowerShell 창에서 `Ctrl+C`를 누릅니다.

### 2.4 테스트 시작 전 초기화

1. 왼쪽 메뉴에서 `TC-4`를 선택합니다.
2. `테스트 큐 비우기`를 누릅니다.
3. 왼쪽 아래 `판정 초기화`를 누릅니다.
4. TC-1로 돌아갑니다.
5. 브라우저 확대율을 100%로 맞춥니다.
6. 다른 탭에서 같은 주소를 열어두지 않습니다. 동일 IndexedDB를 공유하므로 큐 개수가 섞일 수 있습니다.

## 3. 화면에서 결과를 기록하는 방법

각 TC 화면의 아래쪽에는 두 영역이 있습니다.

- `자동 증거`: 앱이 계산하거나 저장소에서 읽은 결과입니다.
- `테스터 판정`: 사람이 화면을 관찰한 뒤 직접 선택하는 최종 체크란입니다.

판정은 다음 기준으로 사용합니다.

| 판정 | 사용 시점 |
| --- | --- |
| `PASS` | 모든 단계와 기대 결과를 직접 확인함 |
| `FAIL` | 한 가지 이상 기대 결과와 다르며 재현됨 |
| `BLOCKED` | 모델·GPU·서버·권한 등 외부 조건이 없어 수행할 수 없음 |
| `PENDING` | 아직 수행하지 않았거나 결론을 내리지 않음 |

`관찰 내용·증거 위치·재현 조건`에는 최소한 브라우저 버전, GPU, 실제 관찰값, 실패 시 마지막 성공 단계를 적습니다. 판정은 브라우저 LocalStorage에 즉시 저장됩니다.

모든 TC가 끝나면 왼쪽 아래의 `JSON 내보내기`와 `Markdown 내보내기`를 각각 눌러 두 파일을 보관합니다.

## 4. TC-1 — 로컬 로드 및 심문방 초기화

### 목적

브라우저 기능 지원 여부, QA 픽스처 기동, 아리아-09 도입 말투를 확인합니다. 실제 WebLLM 모델이 연결된 빌드에서는 같은 시점에 콜드/캐시 부팅도 별도 측정합니다.

### 수행 절차

1. 왼쪽 `TC-1 로컬 로드 및 심문방 초기화`를 선택합니다.
2. `WebGPU`, `IndexedDB`, `Online API` 환경 칸을 읽습니다.
3. WebGPU가 `DETECTED`인지 기록합니다.
4. IndexedDB가 `READY`인지 확인합니다.
5. `아리아-09 기동`을 한 번 누릅니다.
6. 아래 터미널에 다음 특성이 있는 문장이 나타나는지 확인합니다.
   - 화자명이 `ARIA-09`
   - “감독관 아리아-09입니다”
   - “시작합니까?”와 같은 사무적 경어체
7. 문장 아래의 `QA 픽스처 n ms` 값을 메모합니다.
8. `자동 증거` 상태를 확인합니다.

### 기대 결과

- IndexedDB가 `READY`
- 지원 브라우저에서는 WebGPU가 `DETECTED`
- 아리아-09 도입문이 `입니다/합니까` 말투
- QA 픽스처가 오류 없이 기동

### 실제 모델 KPI 교차검증

현재 하네스의 약 350ms 대기는 실제 1.5GB 모델 로드가 아닙니다. 실제 모델이 연결된 배포 빌드에서 다음을 별도로 기록합니다.

1. 개발자 도구를 열고 Console의 VRAM/WebLLM 오류가 없는지 확인합니다.
2. 사이트 데이터와 모델 캐시를 지운 뒤 새로고침합니다.
3. URL 접속 시점부터 첫 대화 표시까지 측정하여 `coldBootTimeMs <= 30000`인지 확인합니다.
4. 캐시를 지우지 않고 다시 새로고침하여 `cachedBootTimeMs <= 3000`인지 확인합니다.
5. 아리아 A 세션에서 고유 질문을 한 뒤 다른 캐릭터 B로 전환합니다.
6. B의 첫 응답에 A의 이름·사실·대화가 섞이지 않는지 확인합니다.

실제 모델이 연결되지 않았다면 메모에 `실제 WebLLM 모델 자산 없음`을 적고 `BLOCKED`를 선택합니다. QA 픽스처 자체만 평가하는 내부 검증이라면 `PASS`를 선택하되, 메모에 `픽스처 범위 PASS`라고 씁니다.

## 5. TC-2 — 모순 문장 지정 및 디버깅 필터

### 목적

실제 브라우저 텍스트 선택이 NPC 말풍선 안에서 캡처되고, 올바른 반박문이 성공하여 과열 게이지가 정확히 30% 감소하는지 확인합니다.

### 수행 절차

1. `TC-2`를 선택합니다.
2. 상단의 현재 게이지를 기록합니다. 최초 실행 기준 50%입니다.
3. NPC 문장에서 숫자와 글자를 포함한 **`3월 10일`만** 왼쪽에서 오른쪽으로 드래그합니다.
4. 마우스 버튼을 놓습니다.
5. `오프셋 … 일치. 반박 콘솔이 열렸습니다.` 메시지를 확인합니다.
6. 반박 콘솔이 열리지 않았다면 빈 곳을 클릭하고 더 정확히 다시 드래그합니다.
7. 입력창의 기본 문장을 그대로 사용하거나 다음 문장을 입력합니다.

```text
3월 15일 공장 가동 및 3월 29일 탈취 기록을 확인하십시오.
```

8. `교정 제출`을 누릅니다.
9. 시스템 메시지에서 다음 세 가지를 확인합니다.
   - `SUCCESS`
   - 일치한 키워드 목록
   - `50% → 20%`처럼 정확히 30% 감소
10. `자동 증거`가 `PASSED`인지 확인합니다.

### 부정 경로 추가 확인

1. 페이지를 새로고침하거나 TC를 다시 엽니다.
2. 네 글자 미만의 짧은 구간을 선택합니다.
3. 반박 콘솔이 열리지 않는지 확인합니다.
4. 정상 말풍선/유저 말풍선 테스트는 현재 하네스 화면에 별도 픽스처가 없으므로 자동 통합 테스트 결과를 함께 참조합니다.

### 실패로 판정할 조건

- NPC 문장을 드래그했는데 콘솔이 열리지 않음
- NPC 말풍선 밖 선택으로 콘솔이 열림
- 키워드 2개 이상인데 검증 실패
- 성공 후 감소량이 30%가 아님
- 피드백 처리 시간이 100ms를 초과함

## 6. TC-3 — 과열 임계치 및 UI 하이재킹

### 목적

과열 임계값에 따라 시각 방해가 단계적으로 강해지고, 입력 유실과 100% 파괴 상태가 관찰되는지 확인합니다.

### 수행 절차

1. `TC-3`을 선택합니다.
2. `80% 주입`을 누릅니다.
3. 다음을 눈으로 확인합니다.
   - 게이지가 80%까지 채워짐
   - 화면에 가로 스캔라인이 표시됨
   - 붉은 오류 팝업이 표시됨
4. `90% 주입`을 누릅니다.
5. `5% 확률 입력 유실 체험` 입력창에 30자 이상 계속 입력합니다.
6. 확률상 삭제를 보지 못했다면 `QA 강제 유실 1회`를 누릅니다.
7. 입력 끝 한 글자가 사라지고 `관찰된 유실 1회` 이상인지 확인합니다.
8. 입력 패널이 원래 위치에서 약간 흔들린 오프셋으로 표시되는지 확인합니다.
9. `100% 주입`을 누릅니다.
10. 붉은 배경과 `SYSTEM COLLAPSE // ANDROID DAMAGED`를 확인합니다.
11. 자동 증거가 `PASSED`가 되려면 100% 도달과 입력 유실 1회가 모두 필요합니다.

### 시간 기반 교차검증

임계치 버튼은 시각 상태를 빠르게 검증하기 위한 QA 주입입니다. 실제 밸런스 검증에서는 버튼을 사용하지 않고 세션을 방치하여 다음을 확인합니다.

- 디버깅 0회일 때 약 60초에 파괴 상태 도달
- 80% 이상에서 팝업이 약 4초마다 생성
- 90% 이상에서 실제 입력 이벤트의 약 5%가 유실

현재 하네스는 실시간 60초 전체 게임 루프 대신 코어 상태 머신의 60초 전이를 자동 테스트합니다. 실제 플레이 타이밍은 배포 게임 루프 연결 전까지 `BLOCKED`로 기록합니다.

## 7. TC-4 — 개인정보 마스킹 및 DPO 캡처

### 목적

원문 개인정보가 저장되지 않고, 마스킹된 `chosen` 값만 IndexedDB 오프라인 큐에 남는지 확인합니다.

1. `TC-4`를 선택합니다.
2. `테스트 큐 비우기`를 누르고 `0건`인지 확인합니다.
3. 기본 입력문에 다음 가짜 값이 있는지 확인합니다.
   - `010-1234-5678`
   - `tester@example.com`
4. `마스킹 후 오프라인 저장`을 누릅니다.
5. 큐가 `1건`으로 증가하는지 확인합니다.
6. 화면의 `chosen (sanitized)` 값을 읽습니다.
7. 다음 토큰이 있는지 확인합니다.
   - `[REDACTED_PHONE]`
   - `[REDACTED_EMAIL]`
8. 원래 전화번호와 이메일이 화면에 남아 있지 않은지 확인합니다.
9. 자동 증거가 `PASSED`인지 확인합니다.

### 주민등록번호 추가 확인

가짜 테스트 값 `970115-1152341`을 입력문에 추가하고 다시 저장합니다. `[REDACTED_RRN]`으로 바뀌어야 합니다. 실존 인물의 번호를 사용하지 마십시오.

### 개발자 도구 교차검증

1. `F12`를 누릅니다.
2. `Application` 탭을 엽니다.
3. 왼쪽에서 `IndexedDB` → `interrogation_station_db` → `tbl_offline_telemetry_queue`를 선택합니다.
4. 최신 행의 `payload.chosen`을 확인합니다.
5. 화면 표와 같은 마스킹 토큰이 있고 원문 PII가 없는지 확인합니다.

## 8. TC-5 — 오프라인 저장 및 네트워크 복구 동기화

### 목적

오프라인일 때 로컬 큐에 저장하고, 온라인 복구 후 전송 성공한 레코드만 큐에서 삭제하는지 확인합니다.

### 수행 절차

1. `TC-5`를 선택합니다.
2. `큐 비우기`를 누릅니다.
3. `Local queue 0`, `QA receiver 0` 또는 현재 누적값, `Network OFFLINE`을 확인합니다.
4. `1. 오프라인 전환`을 누릅니다.
5. `2. 샘플 적재`를 한 번 누릅니다.
6. `Local queue`가 1 증가했는지 확인합니다.
7. 자동 증거가 `PARTIAL`이고 “아직 원격 동기화를 실행하지 않았습니다”라고 표시되는지 확인합니다.
8. `3. 온라인 복구·동기화`를 누릅니다.
9. 다음을 확인합니다.
   - `Local queue`가 0
   - `QA receiver`가 적재 건수만큼 증가
   - 자동 증거가 `PASSED`
   - `전송 1건 / 로컬 잔여 0건`

### 실제 네트워크/Supabase 교차검증

QA receiver는 브라우저 메모리에 있는 가짜 수신기입니다. 실제 AC-3 조건은 Supabase가 연결된 배포 빌드에서 다음과 같이 검증합니다.

1. 개발자 도구 Network 탭에서 `Offline`을 선택하거나 PC 네트워크를 끊습니다.
2. 디버깅 성공으로 DPO 1건을 생성합니다.
3. IndexedDB 큐가 1건 증가하는지 확인합니다.
4. Network 탭을 `No throttling`으로 되돌리거나 네트워크를 연결합니다.
5. `online` 이벤트 직후 실제 텔레메트리 요청의 HTTP 성공 응답을 확인합니다.
6. Supabase에서 같은 `data_id`가 정확히 1건 존재하는지 확인합니다.
7. 성공 응답 이후 로컬 큐가 0건인지 확인합니다.
8. 서버가 실패 응답을 반환할 때는 로컬 큐가 삭제되지 않는지도 확인합니다.

실제 엔드포인트가 없으면 `BLOCKED`로 판정합니다.

## 9. 결과 제출

1. 왼쪽 메뉴에서 모든 TC의 점 색을 확인합니다.
   - 초록: PASS
   - 빨강: FAIL
   - 노랑: BLOCKED
   - 테두리만 있음: PENDING
2. PENDING이 남아 있으면 해당 TC로 돌아가 판정합니다.
3. `JSON 내보내기`를 누릅니다.
4. `Markdown 내보내기`를 누릅니다.
5. 생성된 두 파일을 동일한 테스트 회차 폴더에 보관합니다.
6. 파일명에 빌드 버전과 실행 시각을 추가해도 좋습니다.

권장 제출 폴더 예시는 다음과 같습니다.

    ├── dais-m9-qa-result.md
