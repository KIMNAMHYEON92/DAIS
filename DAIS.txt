1. 시스템 아키텍처 및 핵심 상태 전환 설계 (Core Logic & State)
본 장에서는 1인 개발 환경에서 서버 유지 비용 0원($0)을 달성하고, 로컬 하드웨어(WebGPU) 자원을 극대화하여 구동하기 위한 기술 아키텍처 설계와 개발용 AI 및 테크니컬 아티스트가 즉시 오차 없이 로직화할 수 있는 상태 전이 모델(Finite State Machine)을 정의합니다.

1.1 WebGPU/WebLLM 기반 온디바이스 하이브리드 아키텍처
본 시스템은 유저의 웹 브라우저 리소스(WebGPU)를 활용해 AI 모델을 로컬에서 추론하는 온디바이스(On-device) 구조를 가집니다. 오프라인에서도 모든 핵심 플레이가 가능하며, 네트워크가 연결되었을 때만 익명화된 DPO 학습 데이터셋과 세이브 데이터를 중앙 서버로 비동기 송수신하여 인프라 비용을 극적으로 절감합니다.

1.1.2 데이터 흐름 및 모듈간 역할 명세
1. Game Manager (시스템 컨트롤러): 게임의 전체 생명 주기(오픈, 실행, 종료)와 UI 전환, 상태 변화 및 로컬 DB 기록을 통제하는 중추 컨트롤러입니다.
2. Quantized Model Loader (모델 로더): 4-bit 또는 8-bit로 양자화(Quantized)된 경량 모델(예: Gemma-4-E2B 등)의 가중치를 브라우저 캐시에서 검사하고, 없을 시 CDN에서 비동기 다운로드하여 WebGPU 메모리에 로드합니다.
3. Context Window & Memory Manager (컨텍스트 관리자): 실시간 대화 기록, 페르소나 System Prompt, 압축된 요약본을 조합하여 WebLLM 추론 엔진에 주입할 프롬프트 컨텍스트를 구성하고 실시간으로 스위칭합니다.
4. Drag & Keyboard Input Detector (입력 디텍터): 플레이어가 대화 로그에서 긁은 텍스트의 오프셋 범위(Start index, End index)를 감지하고, 반박 타이핑창에 입력된 텍스트의 유효성을 파악합니다.
5. Telemetry & Data Sync API (중앙 서버 연동): 네트워크 감지기(Network Status Monitor)와 연동하여, 온라인 상태일 때 로컬 버퍼에 쌓인 익명 DPO 데이터 세트({Context, Prompt, Rejected, Chosen})를 Supabase DB로 비동기 벌크 전송(Bulk-Insert)합니다. offline 상태일 때는 IndexedDB에 큐(Queue)로 적재합니다.

1.2 심문실 게임플레이 전체 시퀀스 및 예외 처리 흐름
플레이어가 특정 안드로이드를 심문실로 호출하여 정상/처분 판정을 내리고 최종적으로 서버에 데이터를 전송하기까지의 전체 순차적 동작 흐름(Sequential Logic)입니다.
1.2.1 심문실 핵심 시퀀스 다이어그램
sequenceDiagram


    autonumber


    actor Player as 플레이어 (User)


    participant Game as Game Manager (클라이언트)


    participant Engine as WebLLM 추론 엔진 (로컬 VRAM)


    participant Server as 데이터 수집 서버 (Supabase)


    %% 준비 단계

        Note over Game, Engine: 온디바이스 메모리 스위칭 및 가중치 확인

        Engine-->>Game: LLM 컨텍스트 로드 완료 (VRAM 배정 완료)


    end


    


    %% 루프 단계 (정상 대화 및 모순 발생)

        Game->>Game: 실시간 과열 게이지 (Overclock Gauge) 누적 작동 시작


        Player->>Game: 질문 텍스트 타이핑 입력


        Game->>Engine: [System Prompt] + [대화 이력] + [신규 질문] 전달


        Engine-->>Game: 실시간 토큰 스트리밍 응답 (Char-by-Char 타이핑 연출)


        Game->>Player: UI 화면에 텍스트 렌더링


    end


    %% 에러 검출 및 디버깅 분기


    rect rgb(255, 240, 245)


        Note over Player, Game: 모순 발생 감지 (Hallucination 캡처)


        Player->>Game: 모순 텍스트 영역 마우스 드래그 (Drag Select)


        Game->>Game: 대화 타임스탬프 일시정지 및 반박문(교정 텍스트) 입력 필드 팝업


        Player->>Game: 논리 교정 반박 텍스트 타이핑 입력 후 완료 단축키(Enter) 누름


        Game->>Game: 1차 유효성 검사 (공백 검사, 최소 자수 검사)


    end


    %% 판정 및 보상 처리


    Player->>Game: 최종 정상(Normal) 또는 불량(Defective) 판정 버튼 클릭


    Game->>Game: 판정 결과 대조 (기획 테이블상의 정답 여부 대조)


    alt 판정 성공 (Success)


        Game->>Game: 호감도(Sync Rate) 상승 연출, '크레딧' 재화 지급


        Note over Game, Server: 백그라운드 데이터 수집 처리 (DPO 데이터 가공)


        Game->>Game: 로컬 IndexedDB 세이브 데이터 즉시 갱신


        Game->>Server: [DPO 데이터] 및 [개인정보 필터링 완료된 데이터] 비동기 업로드 (Background)


    else 판정 실패 또는 과열 폭주 방치 (Failure)


        Game->>Game: 과열 게이지 100% 임계점 도달 (UI Hijacking 발생)


        Game->>Game: 데드신 일러스트 출력, 당일 획득 재화 몰수 및 안드로이드 잠금 처리


    end
1.2.2 단계별 예외 처리 (Exception Handling) 및 바운더리 규칙
[예외 1] VRAM 용량 초과 또는 WebGPU 지원 불가 브라우저 환경
* 감지 방법: 클라이언트 실행 초기 단계에서 navigator.gpu 객체 존재 여부 체크 및 requestAdapter() 호출 시 null 리턴 체크. WebLLM 로더가 VRAM 할당 실패 에러(Out of Memory Exception)를 감출 경우 catch 블록에서 제어권을 인계받음.
* 처리 규칙:
   1. WebGPU 미지원 기기: 경고 모달 레이어 출력 후, 클라우드 우회 방식(Fallback API 모드)으로의 전환 버튼을 띄우거나, "VRAM 사양이 낮아 경량 구동 버전(양자화 4-bit)으로 강제 변환합니다."라는 UI 안내 팝업 출력.
   2. 시스템 메모리 여유 공간이 극도로 부족한 경우: 로컬 브라우저 세션 캐시를 초기화하고 기존 대화 컨텍스트 길이를 절반(Context Window = 512 Tokens)으로 엄격히 축소 설정하여 강제 실행.
[예외 2] 백그라운드 데이터 전송 시 네트워크 단절 환경
* 감지 방법: 브라우저 기본 API navigator.onLine 상태 확인 및 주기적인 서버 하트비트(Heartbeat) 응답 타임아웃(3초 초과) 검출.
* 처리 규칙:
   1. 플레이어에게 "네트워크 연결이 끊겼습니다." 등의 게임 진행을 방해하는 팝업을 절대 띄우지 않고, 비동기 전송 큐에 생성된 JSON 데이터를 유보함.
   2. 수집 데이터 세트를 로컬 IndexedDB 내의 임시 백로그 저장 테이블(tbl_offline_telemetry_queue)에 로컬 키값을 부여하여 적재함.
   3. 네트워크 상태가 online으로 복구되는 시점(window.addEventListener('online'))에 백업 큐의 데이터를 최대 5개씩 슬라이싱하여 백그라운드에서 청크(Chunk) 단위로 자동 분할 업로드 및 IndexedDB 비우기 실행.
[예외 3] 드래그 지정의 유효 범위를 벗어난 비정상 인터랙션
* 감지 방법: 유저가 마우스를 놓는 mouseup 이벤트 발생 시, 캡처된 텍스트 범위의 length가 0글자이거나, 단순 개행 문자(\n), 공백 문자( )만 포함하는 경우, 또는 시스템 메시지 필드 영역을 긁었을 경우 검출.
* 처리 규칙:
   1. 드래그 취소로 간주하고 디버깅 입력창 UI 팝업 트리거를 강제로 취소시킴.
   2. 오직 안드로이드 NPC가 실제로 발화한 대화 말풍선 내 텍스트 엘리먼트(css_class: .npc-bubble)에서 발생한 드래그 범위만 유효 오프셋 인덱스로 매칭하여 시스템 변수에 세팅함.

1.3 상태 전이 테이블 및 유한 상태 머신 (FSM)
원활한 게임 루프 제어를 위해 시스템은 3가지 상태 머신에 의해 구동됩니다.


1. GMS (Global Game Manager State): 게임의 전체 씬(Scene) 관리 상태 머신
2. ITSS (Interrogation Turn Session State): 심문실 대화 1턴 내부의 인터랙션 관리 상태 머신
3. APS (Android Persona State): 심문 대상인 안드로이드의 감정/안정성 오염 단계 상태 머신
1.3.1 Global Game Manager State (GMS)
게임 애플리케이션의 최상위 계층을 관리하는 씬 전환 상태 머신입니다.


현재 상태 (Current State)
	전이 타겟 상태 (Next State)
	발생 이벤트 / 전이 조건 (Trigger Event / Condition)
	전이 시 시스템 데이터 변화 및 초기화 규칙
	INIT
	MAIN_LOBBY
	WebGPU 엔진 정상 기동 + 로컬 IndexedDB 로드 완료
	로컬 세이브 데이터 기반 보유 크레딧, 보유 캐릭터 카드 목록 로드 및 UI 렌더링 시작.
	MAIN_LOBBY
	CHAR_SELECT
	플레이어가 '심문 개시' 버튼 클릭
	심문 가능한 안드로이드 캐릭터 카드의 리스트 데이터를 바인딩하여 선택창 활성화.

1.3.2 Interrogation Turn Session State (ITSS)

1.3.4 경계 조건 및 동시성 예외 방어 설계 (Boundary Logic Rule)
1. 스트리밍 중 드래그 차단 규칙


   * NPC가 답변을 생성하여 화면에 타자기 형태로 글자를 하나하나 찍어내는 NPC_STREAMING 상태 동안에는 플레이어의 대화 로그 드래그 기능을 인터페이스 수준에서 물리적으로 완벽 차단합니다.
   * 스트리밍이 실행되는 동안 모든 대화 로그 돔(DOM) 요소의 CSS 속성에 user-select: none; pointer-events: none;을 동적으로 바인딩하고, 완료 후 USER_WAIT 상태로 들어갈 때 해제합니다. 이를 통해 출력이 진행 중인 미완성 텍스트를 강제로 지정하여 데이터셋 분석기가 꼬이는 오류를 원천 차단합니다.


2. 교정 타이핑 중 시간 소진 시 강제 전이


   * 플레이어가 드래그를 완료하여 TYPING_CORRECTION으로 넘어가 타자를 입력하는 동안에도 메인 제한 시간 타이머(Session_Global_Timer)는 정지하지 않고 계속 차감됩니다.
   * 만약 디버깅 완료 전 글로벌 타이머가 0이 된다면, 플레이어의 입력 상태를 강제로 차단하고 즉시 DEATH_SEQUENCE 상태로 강제 전환하여 결과를 연산합니다.

[2단계] 중기 기억 슬롯 (Mid-term Summarized Buffer)

[SUMMARY RESULT OUTPUT]


* 결과 처리: 반환된 한 문장을 Mid_Term_Buffer_Array에 추가하고, 롤링 버퍼의 1~6턴 데이터를 소멸시켜 윈도우 슬롯 400 토큰 공간을 즉각 확보합니다.

* 등록 규칙: 플레이어의 중요한 선택이나 스토리 전환이 발생하는 핵심 교정 이벤트가 성공하는 시점에 한하여, 시스템 단에서 하드코딩된 '장기 앵커 코드'를 트리거하여 업데이트합니다.

3.1.3 드래그 범위 및 문자열 매칭 정밀 공식
* 정확한 매칭 조건 (String Segment Matching):
   1. 유저가 드래그한 선택 문자열 $S_{user}$가 해당 캐릭터의 hallucination_rules[i].erroneous_statement 내에 존재하는지 검증합니다.
   2. 오클릭으로 인한 불쾌감을 방지하기 위해, 에러 키워드 $K_{error}$가 $S_{user}$에 포함되어 있다면 부분 일치도 정상 매칭으로 처리합니다.
   3. 판정 수식: $$\text{Is_Match} = (S_{user} \subseteq S_{error}) \land (\text{Length}(S_{user}) \ge \text{Min_Drag_Length})$$
      * $\text{Min_Drag_Length}$: 기본 설정값 4글자 (공백 제외). 너무 짧은 오동작 드래그 방지용 하한선.

3.3 실시간 과열 게이지(Overclock Gauge) 가감산 공식 및 UI 하이재킹(폭주 방해) 트리거
심문실 내부의 긴박감과 액션 게이머들의 도파민을 자극하는 실시간 제어 게이지 및 폭주 방해 기믹 사양입니다.
3.3.1 과열 게이지($OG$) 연산 및 누적 공식
과열 게이지 $OG$는 $0$에서 $100$ 사이의 실수값을 가지며, 시스템 틱 주기($\Delta t = 0.016s$ / 60 FPS 기준)마다 다음 물리 공식에 의해 실시간 갱신됩니다.


$$OG_{t} = \max\left(0, \min\left(100, OG_{t-\Delta t} + \Delta OG_{passive} + \Delta OG_{event}\right)\right)$$


* 패시브 가산량 ($\Delta OG_{passive}$): $$\Delta OG_{passive} = R_{base} \times M_{state} \times \Delta t$$


   * $R_{base}$: 초당 기본 상승량 (Default: 1.5% / sec)
   * $M_{state}$: 캐릭터 감정 상태(APS 상태) 가중 계수
      * STABLE: $1.0$
      * SUSPICIOUS: $1.2$
      * UNSTABLE: $1.5$
      * OVERCLOCKED: $2.0$ (매우 가파르게 상승)


* 이벤트 가산/감산량 ($\Delta OG_{event}$):


   * 유저의 잘못된 텍스트 입력 전송: $+5%$ (안드로이드의 의심 자극)
   * 대화 로그 내 엉뚱한 텍스트 영역 오클릭/드래그 오류: $+8%$
   * 디버깅 콘솔 반박문 검증 실패: $+15%$
   * 디버깅 콘솔 반박문 검증 성공 (디버깅 완수): $-30%$
   * 최종 판정 전 아리아가 장황한 환각 답변(텍스트 스트리밍)을 출력 중일 때의 초당 추가 증가 가속도: $+1.0%$

3.3.2 과열 단계별 UI 하이재킹(폭주 방해) 연출 트리거 및 제어 규칙
$OG$가 특정 임계값을 넘어설 때, 안드로이드의 논리 소자 폭주로 인해 게임 UI 전체가 플레이어를 직접 공격하고 방해하는 기믹 사양입니다.


graph TD


    OG_Value[실시간 과열 게이지 OG 체크]


    


    OG_Value -->|OG >= 60| Suspicious_Glitch[단계 1: 스캔라인 및 미세 화면 지지직 글리치 효과 활성화]


    OG_Value -->|OG >= 80| Warning_Popups[단계 2: 기술 에러 경고 팝업 레이어 무작위 생성 시작]


    OG_Value -->|OG >= 90| Hard_Hijack[단계 3: 마우스 흔들림 인텐시티 활성화 + 타이핑 백스페이스 오작동]


    OG_Value -->|OG == 100| Death_Seq[단계 4: 강제 파괴 배드엔딩 트리거]
[단계 1] 화면 지지직 글리치 연출 ($OG \ge 60$)
* 연출: 화면 전체에 아날로그 TV 노이즈 같은 가로 스캔라인 필터가 불규칙하게 깜빡입니다.
* 오디오: 지지직하는 0.1초 단위의 짧은 정전기 노이즈 효과음(SFX)이 15% 확률로 대화 중 출력됩니다.
[단계 2] 경고창 팝업 도배 난입 ($OG \ge 80$)
* 기믹: 심문실 화면 중앙 및 주변에 "ERROR: Core Overheat 0x99", "WARNING: Memory Leak" 등 인게임 디자인의 경고창 팝업이 무작위 좌표에 생성됩니다.
* 방해 규칙: 팝업창은 대화 로그나 입력창을 시각적으로 가립니다. 플레이어는 마우스로 팝업 우측 상단의 [X] 버튼을 직접 클릭하여 파괴해야만 가려진 텍스트를 마저 읽거나 인풋 박스 포커싱을 잡을 수 있습니다. 팝업은 4초당 1개꼴로 자동 생성됩니다.
[단계 3] 마우스 하이재킹 & 타이핑 이탈 ($OG \ge 90$)
* 마우스 강제 흔들림 (Shaking Vector): 마우스 포인터의 실제 좌표 $P(x, y)$에 시스템이 강제로 무작위 정현파 오프셋을 가산하여 조준 및 클릭을 방해합니다. $$x_{shaken} = x_{raw} + A_{shake} \times \sin(\omega t) \times \text{Random}(-1, 1)$$ $$y_{shaken} = y_{raw} + A_{shake} \times \cos(\omega t) \times \text{Random}(-1, 1)$$
   * $A_{shake}$ (흔들림 진폭): 최대 15픽셀
   * $\omega$ (진동 주파수): 25Hz
* 타이핑 강제 유실 (Backspacing): 유저가 디버깅 입력 필드에 반박문을 타이핑하는 동안, 시스템은 $5%$의 무작위 확률로 현재 입력된 문자열의 가장 마지막 한 글자를 백스페이스 입력 없이 강제로 소멸시켜 오타를 유발하고 지연을 발생시킵니다.
[단계 4] 시스템 완전 붕괴 ($OG \ge 100$)
* 기믹: 안드로이드의 가슴 혹은 눈 부위 코어가 붉게 폭주하며 화면 전체가 화이트 아웃 처리됩니다.
* 페널티: 당일 획득한 크레딧이 전부 증발하며, 심문 중이던 안드로이드가 파손 플래그(is_damaged: true)와 함께 로비 및 캐릭터 선택창에서 60분간 비활성화(수리 중 상태) 처리됩니다.

4.3.2 5인 교차 검증 합의 산출 공식 (Consensus Mathematics)
중앙 서버는 미검증 풀에 담긴 데이터셋 $D_i$에 대하여 무작위 플레이어들에게 노출시키고, 최종 검수 판정 테이블을 아래 수학적 규칙에 의거해 연산합니다.


* 합의 집합 정의:


   * $V_{i}$ = 데이터 $D_i$에 대해 누적된 총 투표자 집합 ($|V_{i}| = 5$)
   * $v_{i, u} \in {1 \text{ (Pass)}, 0 \text{ (Fail)}}$ : 유저 $u$가 내린 투표값


* 최종 판정 규칙: $$\text{Consensus}(D_i) = \sum_{u \in V_i} v_{i, u}$$


   1. 골드 데이터셋 확정 (Accept): $$\text{Consensus}(D_i) \ge 4$$
      * 5명 중 4명 이상이 유효한 교정이라고 판정함. 데이터의 신뢰도가 80% 이상 확보되었으므로 정식 데이터셋으로 마이그레이션 처리 후 보상 지급.
   2. 어뷰징 쓰레기 데이터 판정 및 즉시 폐기 (Reject): $$\text{Consensus}(D_i) \le 1$$
      * 5명 중 4명 이상이 Fail을 던짐. 의미 없는 문장이나 욕설로 판단되어 데이터 즉각 소멸 처리. 데이터 생산자에게 신뢰 페널티 부과.
   3. 재평가 보류 (Pending): $$\text{Consensus}(D_i) \in {2, 3}$$
      * 합의가 불분명한 경계 영역. 투표자 모수를 추가하기 위해 검수 대기 풀에 잔류시키고 새로운 무작위 검수자 3명에게 다시 노출하도록 가중치 조절.

[Phase 6] MVP 기능 검증 기준 및 QA 체크리스트 (Validation Criteria)
본 장에서는 프로토타입(MVP) 빌드 단계에서 본 게임 기획의 **핵심 재미 요소(도파민 코어 루프)**와 **엔지니어링 특이점(WebGPU 로컬 추론 및 오프라인 DPO 수집)**이 실제 상용화 가능한 수준으로 완벽하게 구현되었는지 검증하기 위한 구체적인 지표, 통과 요건 및 시나리오별 QA 체크리스트를 정의합니다.


________________


6.1 핵심 검증 지표 (Core KPIs for MVP)
본 MVP 테스트 단계에서 달성해야 하는 기술적·기획적 성능 목표치는 다음과 같습니다.
6.1.1 로컬 AI 엔진 성능 지표 (Technical Performance)
* 초기 로딩 속도 (Cold Boot Time): 사용자가 웹 URL에 최초 접속한 후 모델 파일(약 1.5GB) 다운로드 및 WebGPU VRAM 마운트 완료까지의 시간 $\le \mathbf{30}$초 (Wi-Fi 100Mbps 환경 기준). 캐싱 완료 후 재부팅 시간 $\le \mathbf{3}$초.
* 초구 생성 속도 (First Token Latency): 플레이어 질문 전송 엔터 입력 후 첫 번째 답변 글자가 타이핑 렌더링되기 시작할 때까지의 대기 시간 $\le \mathbf{1.2}$초.
* 토큰 생성 처리량 (Inference Throughput): 텍스트 스트리밍 속도 $\ge \mathbf{25}$ tokens/second (RTX 3060 / Apple M1 기준).
* 메모리 안정성 (Memory Leakage): 동일 안드로이드와 30턴 이상의 연속적인 장기 대화를 나눴을 때, 컨텍스트 윈도우 압축 파이프라인 오작동으로 인한 브라우저 VRAM Out of Memory(OOM) 크래시 발생 횟수 0회.
6.1.2 게임플레이 밸런스 지표 (Gameplay Metrics)
* 대화 세션 수명 (Session Duration): 과열 폭주 게이지가 작동하는 상황에서, 숙련된 플레이어가 단 하나의 디버깅도 성공하지 못했을 때 강제 파괴 배드엔딩까지 도달하는 기본 제한 시간 $T_{limit} = \mathbf{60}$초 내외 보장.
* 디버깅 피드백 감쇠 속도: 교정 타이핑 완료 시 과열 게이지 즉각 차감($-30%$) 및 스크린 글리치 오프셋 청소 처리가 프레임 드랍 없이 $0.1$초 내에 즉각 피드백 렌더링될 것.

6.2 MVP 빌드 통과 기준 (Acceptance Criteria)
MVP 마일스톤 빌드가 검수를 통과하여 커뮤니티 알파 테스트로 전환되기 위해 충족해야 할 기능적 인수 조건(Acceptance Criteria)입니다.
[AC-1] 단일 LLM 로컬 구동 및 프롬프트 스위칭
* 조건 1: 브라우저 디버거 콘솔 창에서 로컬 VRAM 마운트 에러 로그가 발생하지 않아야 합니다.
* 조건 2: 안드로이드 A에서 안드로이드 B로 카드를 교체하여 심문을 시작할 때 기존 컨텍스트 잔재가 완전히 가비지 컬렉션(GC)되고 신규 JSON 프로필만 주입되는 것을 메모리 세션 해시를 통해 검증해야 합니다.
[AC-2] 정밀 오프셋 드래그 및 모순 감지 판정
* 조건 1: 플레이어가 마우스 드래그를 완료했을 때, 정확히 안드로이드의 erroneous_statement 텍스트 영역을 포함하여 긁었을 때만 디버깅 콘솔 레이어가 활성화되어야 합니다.
* 조건 2: 빈 공간, 유저 본인의 말풍선, 또는 정상 판정된 문장을 드래그했을 때는 디버깅 콘솔이 절대 열리지 않고 과열 게이지 패널티($+8%$) 가산 연산이 정상 수행되어야 합니다.
[AC-3] PII 비식별화 및 DPO 포맷팅 적재
* 조건 1: 교정 입력창에 한국어 주민등록번호, 전화번호, 이메일을 입력하고 제출했을 때, 로컬 IndexedDB의 tbl_offline_telemetry_queue에 저장된 JSON 페이로드 파일 내의 chosen 필드에는 해당 텍스트가 정확히 [REDACTED_RRN], [REDACTED_PHONE], [REDACTED_EMAIL] 토큰으로 영구 치환되어 기록되어야 합니다.
* 조건 2: 로컬 기기의 인터넷 선을 뽑은 상태(Offline)에서 디버깅 성공 시 IndexedDB 적재가 완료되고, 인터넷 연결을 재개했을 때 Supabase 중앙 DB 테이블로 누락 없이 전송 및 로컬 IndexedDB 버퍼 초기화 처리가 100% 자동 실행되어야 합니다.
